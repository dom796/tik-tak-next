# i18n Design: next-intl with Routing

## Overview

Add Russian/English language support using `next-intl` with URL-based routing. Language is encoded in the URL prefix (`/en/games`, `/ru/games`). Default language is English — `/` redirects to `/en`.

## Stack

- **next-intl v3+** — i18n library for Next.js App Router
- **Middleware** — intercepts requests, detects locale, redirects if missing
- **URL routing** — locale as first URL segment

## Routing Config

```ts
// src/i18n/routing.ts
import { defineRouting } from 'next-intl/routing';

export const routing = defineRouting({
  locales: ['en', 'ru'],
  defaultLocale: 'en'
});
```

## Request Config

```ts
// src/i18n/request.ts
import { getRequestConfig } from 'next-intl/server';
import { hasLocale } from 'next-intl';
import { routing } from './routing';

export default getRequestConfig(async ({ requestLocale }) => {
  const requested = await requestLocale;
  const locale = hasLocale(routing.locales, requested)
    ? requested
    : routing.defaultLocale;

  return {
    locale,
    messages: (await import(`../../messages/${locale}.json`)).default,
  };
});
```

## Navigation Helpers

```ts
// src/i18n/navigation.ts
import { createNavigation } from 'next-intl/navigation';
import { routing } from './routing';

export const { Link, redirect, useRouter, usePathname } = createNavigation(routing);
```

All components and server actions must import `Link`, `redirect`, `useRouter`, `usePathname` from `src/i18n/navigation` — not from `next/navigation` or `next-intl/navigation` directly.

## Middleware

SSE route handlers must be excluded from the middleware matcher, otherwise the middleware will intercept persistent SSE connections and attempt locale redirect, breaking them. Move SSE routes under `/api/` prefix — their handlers already perform direct session checks so they don't need the layout-based auth guard.

```ts
// middleware.ts
import createMiddleware from 'next-intl/middleware';
import { routing } from './src/i18n/routing';

export default createMiddleware(routing);

export const config = {
  matcher: ['/((?!api|_next|.*\\..*).*)']
};
```

SSE routes move to:
- `app/api/games/stream/route.ts`
- `app/api/game/[id]/stream/route.ts`

`kernel/routes.ts` SSE helpers no longer need locale prefix:
```ts
gamesStream: () => '/api/games/stream',
gameStream: (gameId: string) => `/api/game/${gameId}/stream`,
```

## Root Layout

```tsx
// src/app/[locale]/layout.tsx
import { NextIntlClientProvider } from 'next-intl';
import { getMessages } from 'next-intl/server';
import { setRequestLocale } from 'next-intl/server';
import { routing } from '@/i18n/routing';

export function generateStaticParams() {
  return routing.locales.map((locale) => ({ locale }));
}

export default async function LocaleLayout({
  children,
  params,
}: {
  children: React.ReactNode;
  params: Promise<{ locale: string }>;
}) {
  const { locale } = await params;
  setRequestLocale(locale);
  const messages = await getMessages();

  return (
    <html lang={locale}>
      <body ...>
        <NextIntlClientProvider messages={messages}>
          <ThemeProvider>{children}</ThemeProvider>
        </NextIntlClientProvider>
      </body>
    </html>
  );
}
```

Every layout and page that uses the `[locale]` segment must call `setRequestLocale(locale)` before any translations.

## Server Actions and Redirects

Server actions (`sign-in.tsx`, `sign-up.tsx`) call `redirect()` after auth. They must redirect to a locale-aware path. Locale is read from the cookie set by next-intl middleware (`NEXT_LOCALE`):

```ts
import { cookies } from 'next/headers';
import { redirect } from '@/i18n/navigation';

const locale = (await cookies()).get('NEXT_LOCALE')?.value ?? 'en';
redirect('/', { locale });
```

## SSE Routes

SSE route handlers move to `app/api/` (excluded from middleware matcher):
- `app/api/games/stream/route.ts`
- `app/api/game/[id]/stream/route.ts`

`kernel/routes.ts` SSE helpers stay locale-independent:
```ts
gamesStream: () => '/api/games/stream',
gameStream: (gameId: string) => `/api/game/${gameId}/stream`,
```

Auth redirects and links must be locale-aware:

- **Server actions** (`sign-in.tsx`, `sign-up.tsx`, private layout sign-out): use `redirect` from `@/i18n/navigation` — it injects locale automatically from context.
- **Client components** (`sign-in-form.tsx`, `sign-up-form.tsx`): the `BottomLink` component (`src/features/auth/ui/link.tsx`) currently imports `Link` from `next/link` and accepts a `url: string` prop. Update `BottomLink` to import `Link` from `@/i18n/navigation` instead — no other changes needed. The forms continue to pass `routes.signIn()` / `routes.signUp()` as the `url` prop, but those helpers must be updated to return locale-unaware relative paths (`/sign-in`, `/sign-up`) — the locale-aware `Link` will prefix them automatically.

- **Server actions** (`sign-in.tsx`, `sign-up.tsx`, `create-game.tsx`, private layout sign-out): use `redirect` from `@/i18n/navigation` — it injects locale automatically. For `create-game.tsx`, which currently calls `redirect(routes.game(id))`, replace with the locale-aware `redirect` the same way.

`routes.signIn()`, `routes.signUp()`, `routes.game()` in `kernel/routes.ts` return plain relative paths without locale prefix — locale injection is handled by the `Link` / `redirect` from `@/i18n/navigation`.

## LocaleSwitcher Component

```tsx
// src/shared/ui/locale-switcher.tsx
'use client';
import { useLocale } from 'next-intl';
import { useRouter, usePathname } from '@/i18n/navigation';

export function LocaleSwitcher() {
  const locale = useLocale();
  const router = useRouter();
  const pathname = usePathname();

  return (
    <div className="flex gap-1">
      {(['en', 'ru'] as const).map((l) => (
        <button
          key={l}
          onClick={() => router.replace(pathname, { locale: l })}
          className={locale === l ? 'font-bold' : 'opacity-50'}
        >
          {l.toUpperCase()}
        </button>
      ))}
    </div>
  );
}
```

Placed in the private layout header next to the theme toggle. Also added to auth pages.

## next.config.ts

Wrap with `createNextIntlPlugin` for type-safe translation keys:

```ts
import createNextIntlPlugin from 'next-intl/plugin';

const withNextIntl = createNextIntlPlugin('./src/i18n/request.ts');
export default withNextIntl(nextConfig);
```

## Root Layout

The existing `src/app/layout.tsx` must be **deleted**. The new `app/[locale]/layout.tsx` takes over rendering `<html>`, `<body>`, and `ThemeProvider`. Keeping both would produce nested `<html>` elements.

## Implementation Order

1. Install next-intl
2. Update `next.config.ts` with `createNextIntlPlugin`
3. Create `src/i18n/routing.ts`, `request.ts`, `navigation.ts`
4. Create `middleware.ts`
5. Create `messages/en.json` and `messages/ru.json`
6. Delete `src/app/layout.tsx`
7. Restructure `app/` → `app/[locale]/`, move all routes
8. Create `app/[locale]/layout.tsx` — params await, setRequestLocale, NextIntlClientProvider, html/body/ThemeProvider
9. Move SSE routes to `app/api/games/stream/` and `app/api/game/[id]/stream/`
10. Update `kernel/routes.ts` — locale-aware auth routes, API-based SSE routes
11. Replace all hardcoded strings in components (useTranslations / getTranslations)
12. Fix redirects in server actions to be locale-aware
13. Create `LocaleSwitcher` component
14. Add `LocaleSwitcher` to private layout header and auth pages
